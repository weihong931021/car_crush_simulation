# 碰撞模擬重構設計：從「重播＋固定時刻衝量」到「前向模擬＋真實碰撞偵測」

日期：2026-07-20
狀態：草案，待確認
前置：`docs/specs/2026-07-20-scene-bundle-threejs-demo-design.md`（場景包架構，已完成）

## 為什麼要改

現在的模型有一個致命限制：**碰撞被寫死在 `anim_collision`（test1 是 frame 32）**。
車速滑桿只改變碰撞後的衝量大小，不影響「撞不撞得到」。也就是說：

> 把汽車調到 5 km/h，兩車仍然在 frame 32 精準相撞。

這讓 demo 無法回答事故分析最想問的那個問題——「如果慢個幾公里，是不是就閃過了？」
要支援它，物理必須從「重播既有軌跡＋在固定時刻打一發衝量」改成
**「沿路徑前向模擬，真的去偵測兩車有沒有碰到」**。

同時修掉幾個現有簡化帶來的失真（力臂只取前向軸投影 → 正面撞恆無自旋、
接觸點取兩車中心中點 → 力臂不真實）。

## 核心概念：路徑與速度分離

從實際軌跡拆出兩個獨立的東西：

- **路徑（geometry）**：車輛走過的折線，以弧長 `s` 參數化。這是現場證據，不動。
- **速度剖面（speed profile）**：`v(s)`，沿路徑各點的速率。這是可調變數。

車速滑桿的語意從「碰撞瞬間的速度」變成 **「整條速度剖面的縮放係數 k」**：
`v'(s) = k · v(s)`。保留原始的加減速特徵（例如撞擊前的煞車），比設成等速可信。

位置隨時間由積分得到：`t(s) = ∫ ds / v'(s)`。改變 k 會改變兩車抵達衝突區的**時間差**
——這正是「早一點／晚一點就閃過了」的物理本質。

## 模擬流程

```text
scene.json + trajectory.json
    ↓ 抽路徑 + 速度剖面（每台 collider）
    ↓ 套用使用者的 k（車速滑桿）
以固定步長 dt 前向積分：
    每步 → 兩車位置、heading（路徑切線）、速度
         → 以 OBB（車長×車寬，繞 heading）做重疊偵測
    ├─ 有重疊 → 二分細化撞擊時刻 → 用當下真實速度算衝量 → 轉自由體滑行
    └─ 全程無重疊 → 記錄最小間距與發生時刻（近距未撞）
```

### 1. 碰撞偵測

每台車視為 OBB（oriented bounding box）：`length_m × width_m`，繞 `heading` 旋轉。
用分離軸定理（SAT）：2D 兩個矩形只需檢查 4 個軸（各自的兩個邊法向），全都重疊才算碰撞。

- 首個重疊步 → 在前一步（無重疊）與該步之間**二分細化**到 sub-frame 精度
- 未撞時同步計算「兩矩形最短距離」，取全程最小值當作 **miss distance**

`width_m` 是新欄位，需加進 `scene.json` 的 vehicle（現在只有 `length_m`）。

### 2. 衝量與自旋（修掉現有簡化）

在偵測到的實際撞擊時刻：

- **接觸點**：取兩 OBB 重疊區域的中心（或最深穿透點），不再用「兩車中心中點」
- **法向量**：取分離軸上穿透最淺的那個軸（SAT 的標準副產品），比「中心連線」符合實際碰撞面
- **力臂**：`r = 接觸點 − 車輛質心`，**保留完整的 2D 向量**，不再只取前向軸投影
  → 偏心正面撞（真實世界最常見的 spin 來源）從此會產生自旋
- 其餘沿用現行公式：`j = -(1+e)·v_rel,n / (1/m₁ + 1/m₂)`、`τ = (r × J)_y`、`I = m·L²/12`

`OMEGA_MAX` 上限保留為防呆，但正常情況不該再被觸發（現在 test1 是靠上限決定轉量，
代表力臂模型高估了）。上限被觸發時應在 console 明確警告。

### 3. 碰後行為

沿用現有的摩擦滑行積分（`frictionSlide`），但角速度衰減改為與線速度**同步的比例衰減**
（車停 = 轉停，物理上一致），而非目前的獨立線性衰減。

### 4. 未碰撞時的行為

兩車各自沿原路徑走完全程（路徑是證據，速度才是變數）。UI 明確標示：

> 未發生碰撞 · 最近距離 1.24 m（t = 2.4 s）

## 功能：臨界速度求解

有了(1)之後幾乎免費：對其中一台的 k 做二分搜尋，找出「撞 / 不撞」的分界。

```text
汽車實際 20 km/h → 碰撞
汽車 ≤ 13.4 km/h → 閃過（最近距離 0.05 m）
∴ 「若汽車再慢 6.6 km/h 即可避免此事故」
```

- 對兩台各求一次，得到兩個獨立結論
- 單調性需驗證：理論上速度越低越晚到、越可能閃過，但若路徑交會兩次則可能非單調
  → 實作先做粗掃（例如 k ∈ [0.2, 1.5] 掃 40 點）確認單調再二分，非單調則回報區間
- UI：一顆「求臨界速度」按鈕，結果以文字呈現；滑桿上可標出臨界點

這是整個 demo 對事故分析最有說服力的輸出，優先度高於畫面再美化。

## 未來擴充：主動煞車（AEB）模擬（2026-07-24 討論，未實作）

核心改動：把「照剖面播放」改成「剖面＝駕駛意圖，煞車可蓋過它」。

- simulate() 每台車維護當前速度 v，每步 `ds = v·dt` 積分；無煞車時 v 跟隨剖面
  v(s)×k，煞車時 `v = max(0, v − a·dt)`（a ≈ 6–8 m/s²，即 μg）
- 觸發條件：每步算 TTC（兩車 OBB 距離 ÷ 接近速度）或距離門檻，低於門檻＋反應延遲後開始煞
- 路徑不變：煞車只改縱向速度，path.js / obb.js 不動
- 架構：新開 `controller.js` 放觸發＋煞車邏輯，simulate() 加可選 hook，UI 加開關；
  參數三個：觸發 TTC、反應延遲、減速度

### 煞車參數自動求解（同 solve.js 手法：粗掃＋二分）

每組（煞車時機, 力度）跑一次 simulate()，毫秒級，瀏覽器可算：

- **1D 最實用**：力度固定物理極限（μg ≈ 7 m/s²），二分搜「最晚煞車點」→
  輸出「最遲 t=X s 煞車可避撞」，事故分析最有說服力
- **2D 全圖**：時機 × 力度掃格（40×40 = 1600 次模擬，秒級跑完）→
  可避撞區域邊界＝每個時間點最少需要的減速度
- **注意非單調**：煞車可能反而把自己送進衝突窗（同 slowerK/fasterK 洞見），
  必須粗掃找翻轉點再精修，不能假設單一臨界值

## 順帶要修的資料正確性

Blender 實測（2026-07-20）發現宣告值與模型不符，且**物理直接吃這些數字**：

| 項目 | scene.json 宣告 | 模型實測 | 真實車輛 |
| --- | --- | --- | --- |
| car 車長 | 3.8 m | 4.77 m | Model 3 = 4.69 m |
| moto 車長 | 1.7 m | 1.42 m | 一般速克達 ≈ 1.8 m |

- `length_m` 應為**真實車輛尺寸**（碰撞偵測、轉動慣量、力臂全靠它）
- GLB 依 `length_m` **scale-to-length**（現在完全沒縮放，畫面尺寸是模型原始大小）
- 新增 `width_m`（碰撞偵測必需）
- 車種預設值進 `tools/build_scene.py` 的 `CLASS_DEFAULTS`

模型前方軸向已於 commit `3b8b3e1` 修正（car 偏 11°、moto 偏 234°）。

## 架構配置

沿用現有分層，全部新邏輯放純函式層：

```text
threejs/lib/
├── path.js          [新] 折線弧長參數化、速度剖面抽取、t↔s 查詢
├── obb.js           [新] SAT 重疊偵測、最短距離、穿透深度與法向
├── simulate.js      [新] 前向積分主迴圈 → {collided, impactTime, contact, minGap, wps}
├── solve.js         [新] 臨界速度掃描 + 二分
├── physics.js       [改] 衝量改吃真實接觸點與法向；自旋比例衰減
├── waypoints.js     [改] 輸出路徑與速度剖面，而非直接輸出碰前 waypoints
└── interp.js        [不變]
```

- 全部不得 import three.js（維持 `node --test` 可測）
- `main.js` 只負責把模擬結果畫出來與顯示結論文字

## 第二階段：Blender 消費端

模擬結果是一份確定的軌跡，Blender 端因此變單純：讀同一份 `scene.json` ＋ 模擬輸出，
自動搭建渲染場景（地面貼圖、車輛匯入與縮放、逐幀 keyframe），輸出出版用高品質畫面。
與 Three.js demo 同源，不會兩邊各算一套物理。

## 驗收標準

1. test1 以原始車速模擬 → 仍在原碰撞點附近發生碰撞（與現有畫面一致，容許 sub-frame 差異）
2. 汽車降速至臨界值以下 → 明確顯示未碰撞與最近距離，兩車各自走完路徑
3. 臨界速度求解對 test1 給出具體數字，且該數字上下 ±1 km/h 分別為撞／不撞
4. 偏心正面撞（合成測資）產生自旋，不再恆為零
5. 車輛畫面尺寸符合 `length_m`，OBB 與畫面外觀一致（可用頂視圖疊框驗證）
6. `node --test` 涵蓋：SAT 重疊與分離、最短距離、弧長參數化往返、臨界速度單調性

## 實作補記（2026-07-21〜24，與原設計的差異）

1. **軌跡淨化管線**（原設計未涵蓋，實作中因呈現需求逐步定案）：
   `smoothPoints → trimFrozenTail → rdpSimplify(ε=0.06m) → refineAnchors(≤12°) →
   projectToPath → limitAcceleration → extendPoints`。
   關鍵原則＝**幾何/時序分離**：RDP 折線只當直線化中心線，每個樣本點投影上去、
   自己的 t 保留。管線變體對照（test1, k=1）：

   | 變體 | impact | solve（汽車） |
   | --- | --- | --- |
   | 不動幾何（時序基準） | 8.42s | 0.693/null |
   | **投影版（採用）** | **8.39s** | 0.693/null |
   | 抽稀+樣條（舊） | 7.90s | 0.676/1.927 |
   | 純 RDP（時序被粗化） | 10.03s | 0.836/null |

   人工標記碰撞幀 ≈8.5s——投影版最貼近。教訓兩則：ε=0.15 會吃掉真彎（sagitta ~0.2m）；
   純 RDP 丟時間戳＝粗化速度剖面（kA=0.6 的 minGap 0.66→1.48m）。
2. **車輛出現時間 `startT`**（原設計遺漏）：軌跡不從 t=0 開始的車（test1 機車 6.3s
   進場）不得提早出發，否則先騎到衝突點「停著等撞」。出現前不前進、不參與偵測、
   不輸出樣本。
3. **運動學約束**（原設計只有碰後物理）：車身朝向轉向率上限
   `min(0.6v+0.15, a_lat_max/v)`（消蠕行擺頭「飄」＋高速抓地）；縱向加減速
   `a≤3.0 / b≤7.5 m/s²` 前向-後向掃描（位置不動、只重分配段耗時）。
4. **臨界速度→安全區間**：實作證實交會事故為 false→true→false 安全窗（見上方
   「臨界速度求解」的單調性警告成真），介面改為 `solveSafeSpeeds` 回傳
   transitions/safeIntervals/slowerK/fasterK。
5. **呈現決定**：播放至碰撞瞬間截止（內部會議）；渲染＝明亮＋單一主光源＋清楚影子
   （ACES/IBL 否決）；路徑顯示實錄段實線、外插段虛線。
6. **資料陷阱補充**：追蹤器位置在碰前 ~0.5s 凍結（trimFrozenTail 的由來）；
   MotoCollider/CarCollider 是模型層級父節點，只能拔 mesh 引用或以 registry
   `hide` 清單隱藏，不可砍子樹。

## 已確認的決定（2026-07-20）

1. **車速滑桿語意 = 速度剖面縮放 k**。保留原始加減速特徵，滑桿顯示的 km/h 為
   「碰撞前參考點的速率」，內部換算成 k。
2. **臨界速度兩台都求並列出**，且在最近距離發生的時刻畫出兩車間距標註線。
3. **全面重構**，不做過渡並存；現有 test1 行為當回歸基準保護。
4. `width_m` 由車種預設表提供（Car 1.8 / SUV 1.9 / Van 2.0 / Truck 2.5 / Bus 2.55 /
   Two_Wheeler 0.7 m），可在 scene.json 逐場景覆寫。
